## Context

В Trade Rush уже есть доменный слой команд (`TeamService`, `Team`, `TeamRepository`) и UI управления командами через `TeamManagementBlock`. Существующий `TeamService.listTeams()` уже возвращает команды в порядке рейтинга: по очкам от большего к меньшему, затем по имени без учёта регистра. Это делает книгу рейтинга в первую очередь client/server UI-фичей, а не новой доменной логикой.

Vision проекта описывает книгу рейтинга как системный предмет, который игрок берёт в руку и открывает для просмотра списка команд и очков. Книга не должна подменять UI управления командами: создание, вступление, выход, переименование и удаление остаются в `TeamManagementBlock` и чат-командах как резервном/admin/debug пути.

## Goals / Non-Goals

**Goals:**

- Добавить первый отдельный non-block item мода: `TeamRatingBook` с ID `trade-rush:team_rating_book`.
- Открывать read-only UI рейтинга при использовании книги в руке.
- Получать актуальные данные команд на сервере в момент открытия книги и автоматически отправлять обновлённый snapshot в открытую книгу при изменении состояния команд.
- Переиспользовать `TeamService.listTeams()` как единый источник порядка команд и очков.
- Показать игроку место, название команды и количество очков.
- Делить длинный рейтинг на страницы и давать игроку book-like перелистывание страниц.
- Сделать книгу нестакаемой и доступной через creative/admin workflow.
- Обновить README кратким описанием предмета.

**Non-Goals:**

- Не реализовывать бесконечный сундук на спавне для выдачи книги.
- Не добавлять action-кнопки, поля ввода или другие интерактивные действия управления/обновления в UI книги рейтинга.
- Не добавлять в книгу действия управления командами.
- Не использовать бесконечный scroll как основной UX книги, если можно сделать book-like страницы.
- Не менять правила начисления очков, хранения команд или формат сохранений.
- Не добавлять новую внешнюю зависимость.

## Decisions

### Решение: добавить отдельный registry слой для предметов

Сейчас регистрация `TeamManagementBlock` и его block item живёт в `TradeRushBlocks`, но книга рейтинга — первый самостоятельный non-block item. Для неё лучше добавить отдельный `TradeRushItems`, который регистрируется из `TradeRush.onInitialize()` рядом с `TradeRushBlocks.register()`.

`TeamRatingBook` должен использовать ID `trade-rush:team_rating_book`, `Item.Properties().stacksTo(1)` и подходящий creative tab для системных/utility предметов. Если exact имя creative tab отличается в mappings Minecraft `26.1.2`, implementation должна выбрать ближайшую подходящую tab без изменения поведения книги.

Рассмотренные альтернативы:

- Добавить книгу в `TradeRushBlocks`: отклонено, потому что это не блок и такой файл быстро станет смешивать разные типы registry.
- Не добавлять отдельный registry класс до появления второго предмета: возможно, но менее чисто, так как книга уже задаёт паттерн для будущих системных предметов вроде книги товаров.

### Решение: использовать отдельный read-only book UI, а не `TeamManagementScreen`

`TeamManagementScreen` содержит управление командами, выбранную команду, участников, формы ввода и mutating actions. Книга рейтинга должна быть проще и безопаснее: только список мест, команд и очков.

Нужно добавить отдельные menu/screen/networking классы для рейтинга, например под пакетами `platform.ui.rating` и `client.ui.rating`. Экран должен визуально быть book-like: компактный заголовок, строки рейтинга и постраничное отображение длинного списка. Важнее сохранить простоту и читаемость, чем копировать весь vanilla written book behavior.

Рассмотренные альтернативы:

- Переиспользовать `TeamManagementScreen` в read-only режиме: отклонено, потому что management UI визуально и функционально перегружен для книги.
- Генерировать vanilla written book pages/NBT: отклонено для первой версии, потому что данные динамические, сервер должен формировать snapshot при открытии, а запись данных в item NBT может создать лишнее состояние и edge cases.

### Решение: длинный рейтинг отображается страницами, а не бесконечным списком

Книга рейтинга должна ощущаться как книга. Если количество команд превышает вместимость одной страницы, `RatingBookScreen` должен разбивать строки рейтинга на страницы и давать игроку перейти к следующей/предыдущей странице. Ванильные book-like элементы перелистывания страниц допустимы как navigation controls: они не являются `Refresh` или action-кнопками управления командами.

При получении нового snapshot через server push экран должен пересчитать страницы. Если текущая страница после обновления выходит за пределы нового количества страниц, экран должен безопасно перейти на последнюю доступную страницу или первую страницу для empty state.

Рассмотренные альтернативы:

- Вертикальный scroll внутри книги: технически проще, но менее аутентичен для книжного UI.
- Обрезать список до видимых команд: отклонено, потому что игрок должен иметь доступ ко всему рейтингу.
- Увеличивать окно/уменьшать шрифт под весь список: отклонено, потому что ломает читаемость и зависит от GUI scale.

### Решение: snapshot строится при открытии книги и при server-side изменении команд

При `use` предмета server-side игроку открывается menu/screen, затем сервер отправляет clientbound payload со snapshot рейтинга. Snapshot строится из `TradeRush.runtime().teamService().listTeams()` и преобразуется в строки рейтинга с обычной нумерацией `1..N`.

Открытый экран хранит полученный snapshot. Если после открытия изменяется server-side состояние команд — например, команда получает очки через `TeamService.awardPoints`, команда создаётся, переименовывается или удаляется — сервер должен отправить новый snapshot в открытые UI, которые зависят от списка команд/очков. Для книги рейтинга это означает автоматическое обновление отображаемого рейтинга без кнопок и без действий игрока.

Рассмотренные альтернативы:

- Обновлять только через закрытие и повторное открытие: отклонено, потому что открытая книга должна отражать изменения очков автоматически.
- Добавить ручную кнопку обновления: отклонено после уточнения требования; книга должна быть без кнопок.
- Client-side polling/таймер: отклонено, потому что сервер уже знает момент изменения состояния и может отправить push-обновление.

### Решение: использовать `TeamService.onStateChanged` как триггер для UI broadcaster

В текущем коде `TeamService` уже вызывает `onStateChanged.run()` после изменения команд: create, join, leave, award points, rename и delete. Сейчас в `TradeRushRuntime` этот callback указывает на `saveTeamsSafely`. Для автообновления UI нужно заменить его на runtime-level метод вроде `onTeamsChanged`, который сначала сохраняет состояние команд, а затем уведомляет platform-level broadcaster.

Broadcaster не должен жить в доменном слое. Он должен находиться в platform/runtime слое, иметь доступ к `MinecraftServer`, пройти по online players и отправить свежие snapshots только тем игрокам, у которых открыт зависимый menu, например `RatingBookMenu`. Если открытый `TeamManagementMenu` тоже отображает командные очки, broadcaster может обновлять и его через существующий `TeamManagementNetworking`/snapshot path, чтобы все открытые UI, зависящие от командного состояния, получали согласованные данные.

Такой подход не требует ручных subscriptions/unsubscriptions: закрытие UI автоматически означает, что `player.containerMenu` больше не является соответствующим menu.

Рассмотренные альтернативы:

- Хранить список подписчиков при открытии книги: отклонено, потому что нужно аккуратно отписывать игроков при закрытии, дисконнекте и смене dimension/menu.
- Вызывать networking напрямую из `TeamService`: отклонено, потому что доменный слой не должен зависеть от Minecraft/Fabric UI.
- Ввести полноценный typed domain event bus: архитектурно чище для будущего, но тяжелее первой реализации; текущего `onStateChanged` достаточно как coarse-grained сигнала.

### Решение: переиспользовать порядок `TeamService.listTeams()` и не вводить специальные правила tie-ranking

`TeamService.listTeams()` уже сортирует команды по score descending, затем по имени. Книга должна использовать этот порядок напрямую и присваивать обычные места по позиции строки: первая строка — место 1, вторая — место 2 и так далее.

Если у двух команд одинаковые очки, они всё равно получают разные последовательные места. Специальные shared ranks не нужны.

Рассмотренные альтернативы:

- Дублировать сортировку в UI/snapshot builder: отклонено, потому что создаёт риск расхождения с командой `/traderush team list` и `TeamManagementSnapshots`.
- Делать одинаковые места при равных очках: отклонено по продуктовому решению; обычная нумерация проще и достаточна.

### Решение: payload должен быть отдельным и минимальным

Для книги достаточно clientbound snapshot вроде `TeamRatingBookStatePayload` / `TeamLeaderboardSnapshot`, содержащего список строк: `place`, `teamName`, `score`. Serverbound payload для книги на первой итерации не нужен, потому что UI книги не содержит кнопок и не отправляет действий.

Client networking должен, как и `TeamManagementClientNetworking`, уметь принять snapshot payload до полной инициализации screen и применить его, когда `RatingBookScreen` станет активным. При последующих push-обновлениях тот же clientbound payload должен заменить текущий snapshot экрана.

Рассмотренные альтернативы:

- Переиспользовать `TeamManagementSnapshot`: возможно технически, но он содержит лишние поля (`selectedTeamId`, `currentTeamId`, members) и связан с management UI.
- Отправлять plain text pages: проще, но ухудшает future reuse для картины рейтинга и затрудняет layout/тестирование.

## Risks / Trade-offs

- [Fabric/Minecraft API для `Item.use`, menu opening data и creative tabs может отличаться в `26.1.2`] → При реализации проверить актуальные mappings и следовать уже работающему паттерну `TeamManagementBlock`/`TradeRushMenus`.
- [Payload может прийти на клиент до создания screen] → Использовать pending-state pattern, аналогичный `TeamManagementScreen.receiveState`.
- [Частые изменения очков могут отправлять много snapshot-пакетов] → В первой версии отправлять только игрокам с открытыми зависимыми menu; при необходимости позднее добавить debounce/dirty flag на server tick.
- [Большое количество команд может не помещаться на одном экране] → Делить рейтинг на страницы, показывать индикатор текущей страницы и не терять доступ к командам за пределами первой страницы.
- [Runtime может быть недоступен в edge case] → Обработать ошибку без краша клиента/сервера, например показать empty/error state или не открыть экран с понятным сообщением.
- [Дублирование понятия рейтинга] → Не добавлять отдельную сортировку; использовать `TeamService.listTeams()`.

## Migration Plan

1. Добавить регистрацию предмета, menu, payload и screen без изменения existing save format.
2. Добавить read-only snapshot builder поверх существующего `TeamService.listTeams()`.
3. Добавить assets/localization/README.
4. Rollback безопасен: удаление книги и её UI не затрагивает сохранённые команды и очки.

## Open Questions

Нет открытых продуктовых вопросов. Приняты решения: ID `trade-rush:team_rating_book`, обычная нумерация мест, без spawn chest в этой change, книга доступна через creative/admin workflow, UI книги не содержит action-кнопок/полей ввода, длинный рейтинг отображается страницами, данные обновляются при открытии и автоматически через server push при изменении командного состояния.
