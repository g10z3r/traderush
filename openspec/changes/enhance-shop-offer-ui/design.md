## Context

Магазин уже реализован как Fabric block с серверным открытием `ShopOffersMenu` и отправкой `ShopOffersPayload` клиенту через `ShopNetworking.sendOffers(...)`. Клиентский экран `ShopOffersScreen` рисует левую колонку заказов, правую детальную панель, требуемые `ItemStack`-миниатюры и кнопку `Trade`.

Временные заказы представлены доменной моделью `TimedActiveOffer`, где уже хранится `endsAtTick`, но это значение не попадает в клиентский snapshot магазина. Поэтому клиент сейчас не может показать countdown до окончания заказа. При этом `GuiGraphicsExtractor` уже поддерживает vanilla item tooltip через `setTooltipForNextFrame(font, stack, mouseX, mouseY)`, а значит tooltip можно добавить без новых зависимостей и без кастомного tooltip-рендера.

Отдельный связанный риск: текущий `ShopNetworking.handleTrade(...)` удаляет предметы из инвентаря до проверки активного заказа. Если timed-заказ истёк в открытом UI, игрок может потерять предметы без начисления очков. Эта проверка должна быть переставлена до списания предметов.

## Goals / Non-Goals

**Goals:**

- Передать клиенту данные, достаточные для локального countdown временных заказов.
- Показывать countdown в левой строке заказа и в правой панели выбранного заказа, используя один источник расчёта остатка.
- Не показывать countdown для limited-заказов.
- Показывать стандартный vanilla tooltip для требуемых предметов/блоков при hover по иконкам в списке и детальной панели.
- Предотвратить списание предметов, если заказ уже не активен на сервере.

**Non-Goals:**

- Не менять механику генерации, выбора награды или длительности заказов.
- Не добавлять периодическую сетевую синхронизацию или автоматическое обновление списка после истечения заказа.
- Не сохранять active offers между перезапусками сервера: текущий runtime хранит их in-memory, и это не меняется в рамках этой правки.
- Не менять визуальный стиль магазина за пределами необходимых таймеров и tooltip.

## Decisions

### Передавать `expiresAtTick` на уровне `ShopOfferEntry`

`ShopOfferEntry` должен включать поле, например `expiresAtTick`, где для timed-заказов хранится `TimedActiveOffer.endsAtTick`, а для limited-заказов используется sentinel-значение `-1L` или аналогичный маркер отсутствия истечения. Для доступа к значению в домене нужно добавить getter в `TimedActiveOffer`.

**Почему так:** текущий `ShopOfferEntry` уже не является чисто статическим описанием `Offer`: он содержит `fixedReward`, который берётся из конкретного `ActiveOffer`. Поэтому expiry — это такой же active-state для отображения строки UI магазина. Хранение `expiresAtTick` рядом с `fixedReward` делает entry самодостаточным для рендера, упрощает клиентский код и убирает риск рассинхрона между `offers` и отдельной map timer metadata.

**Альтернатива:** передавать timer metadata отдельно в `ShopOffersPayload` как `offerId -> expiresAtTick`. Это сохраняет `ShopOfferEntry` без изменений, но добавляет lookup на клиенте и создаёт дополнительные инварианты: timer metadata должна существовать для каждого timed entry и не должна ссылаться на отсутствующие offers. Для текущей модели UI это избыточно.

### Передавать `serverTick` в `ShopOffersPayload`

`ShopOffersPayload` должен включать `serverTick`, равный `player.level().getGameTime()` в момент сборки offer snapshot. Клиент при `applyPayload(...)` сохраняет `serverTickAtSnapshot` и локальный `clientTickAtSnapshot`, а при рендере оценивает текущий серверный тик как:

```text
estimatedServerTick = serverTickAtSnapshot + (clientCurrentTick - clientTickAtSnapshot)
remainingTicks = max(0, entry.expiresAtTick - estimatedServerTick)
```

**Почему так:** countdown остаётся плавным и синхронизированным между левым и правым блоками без сетевого пакета каждую секунду.

**Альтернатива:** передавать только `remainingTicks`. После сетевой задержки и времени, прошедшего до первого рендера, таймер сразу начнёт отставать, если не привязать его к исходному server tick.

### Один helper для форматирования таймера

`ShopOffersScreen` должен иметь общий helper для:

- проверки, является ли entry timed с валидным expiry;
- расчёта оставшегося времени;
- форматирования строки `H:MM:SS` или `M:SS`;
- определения состояния истечения для отключения `Trade`.

**Почему так:** левый список и правая панель должны показывать синхронизированное значение и одинаково обрабатывать `Expired`.

### Tooltip через vanilla `GuiGraphicsExtractor`

При рисовании каждой иконки требуемого `ItemStack` нужно проверять попадание `mouseX/mouseY` в прямоугольник иконки и вызывать:

```java
graphics.setTooltipForNextFrame(font, stack, mouseX, mouseY);
```

Для этого методы рисования иконок должны получать координаты мыши: `drawRowContent(...)` и `drawItemIcons(...)` либо возвращать/использовать общий helper для draw+hover.

**Почему так:** это даёт стандартное поведение Minecraft без ручного составления localized name/description и без риска отличаться от inventory tooltip.

### Серверная проверка активности до удаления предметов

`ShopNetworking.handleTrade(...)` должен сначала найти active offer через `findAcceptingActiveOffer(...)`, а только затем проверять и удалять предметы. При отсутствии активного заказа сервер должен вернуть ошибку без изменения инвентаря.

**Почему так:** клиентский countdown и disabled button улучшают UX, но сервер остаётся источником истины и обязан защищать инвентарь при гонках/задержках/устаревшем UI.

## Risks / Trade-offs

- [Риск] Countdown основан на локальной оценке server tick и может не учитывать сильные лаги/паузу клиента. → [Mitigation] Использовать server tick snapshot как базу; серверная проверка остаётся authoritative.
- [Риск] После истечения заказа UI покажет `Expired`, но новый active offer не появится автоматически. → [Mitigation] Это осознанный non-goal; пользователь может закрыть/открыть магазин, а auto-refresh можно добавить отдельной change.
- [Риск] Расширение `ShopOfferEntry` и `ShopOffersPayload` меняет сетевой codec. → [Mitigation] Client/server находятся в одном моде и должны обновляться вместе; порядок read/write должен быть изменён синхронно.
- [Риск] Tooltip может перекрываться с row buttons. → [Mitigation] Вызывать `setTooltipForNextFrame` после отрисовки предметов и использовать актуальные координаты мыши из `extractRenderState(...)`.

## Migration Plan

- Миграции сохранённых данных не требуется.
- Сетевой формат `ShopOffersPayload` меняется только внутри client/server одного мода.
- При rollback достаточно вернуть предыдущую версию мода; persistent state не меняется.

## Open Questions
