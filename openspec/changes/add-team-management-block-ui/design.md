## Контекст

В Trade Rush уже есть доменный слой управления командами (`TeamService`, `TeamRepository`, `Team`, `TeamError`) и серверные чат-команды (`TeamCommand`) для CRUD-подобных операций. Состояние команд хранится в памяти и сохраняется через `TradeRushRuntime`/`TradeRushPersistence`. Клиентская часть сейчас почти пустая; блоки, menu/screen и networking для UI ещё не зарегистрированы.

По vision проекта игроки должны управлять командами через специальный блок на спавне, а не через чат. Это сквозная фича: она затрагивает регистрацию блока/предмета, серверный поток взаимодействия, client-side экран, сетевые payloads и существующий доменный слой команд.

## Цели / Не-цели

**Цели:**

- Добавить специальный блок управления командами, взаимодействие с которым открывает UI.
- Показать в UI список команд, очки, количество участников и участников выбранной команды.
- Позволить игроку через UI создавать команды, вступать, выходить, удалять пустые команды и переименовывать пустые команды.
- Оставить сервер авторитетным: все изменения состояния команд выполняются на сервере через `TeamService`.
- Переиспользовать существующие правила валидации и сохранения состояния команд.
- Сохранить существующие чат-команды как резервный, административный и отладочный путь.

**Не-цели:**

- Не реализовывать систему капитанов, владельцев команд или сложные права доступа.
- Не добавлять `force delete` в обычный UI.
- Не реализовывать удаление или переименование непустых команд через UI.
- Не вводить новую внешнюю зависимость.
- Не блокировать магазин/игровые действия для игроков без команды в рамках этой change; это отдельная игровая логика.
- Не доводить финальную модель/текстуру блока до полированного состояния сверх минимальных assets, нужных для работы.

## Решения

### Решение: назвать блок в коде `TeamManagementBlock`

В vision используется термин «командный блок», но в Minecraft уже есть vanilla Command Block. В коде и переводах лучше использовать нейтральное имя вроде `Team Management Block` / «Блок управления командами», чтобы избежать путаницы с vanilla mechanics.

Рассмотренные альтернативы:

- `CommandBlock`: отклонено из-за конфликта смысла с vanilla Command Block.
- `TeamTerminal`/`TeamBoard`: подходят для flavor-названия, но `TeamManagementBlock` лучше описывает роль в коде.

### Решение: UI вызывает `TeamService`, а не дублирует логику команд

`TeamService` уже содержит правила создания, вступления, выхода, списка и удаления команд, а также запускает сохранение через `onStateChanged`. UI actions должны идти через этот же сервис, чтобы чат-команды и UI имели одинаковые ошибки, ограничения и поведение сохранения.

Если переименование команды отсутствует в текущем API, нужно добавить доменный метод `renameTeam` с теми же правилами имени, что и `createTeam`, плюс проверкой уникальности и ограничением на пустую команду для UI-path.

Рассмотренные альтернативы:

- Вызывать Brigadier/chat commands из UI: отклонено, потому что это связывает UI с текстовым интерфейсом и усложняет обработку результатов.
- Реализовать отдельную UI-specific логику: отклонено, потому что это создаст рассинхронизацию правил.

### Решение: expose rename через чат-команду

Так как `renameTeam` становится частью доменного `TeamService`, операция переименования не должна оставаться доступной только через UI. Chat-command layer должен получить ветку `/traderush team rename` и alias `/tr team rename`, которая вызывает тот же `TeamService.renameTeam` и показывает результат через общий `TeamMessages.renamed`/`TeamMessages.componentFor`.

Команда должна сохранять те же доменные ограничения первой версии: переименование разрешено только для пустых команд, новое имя trim-ится, проходит проверку длины и уникальности, id/очки/участники команды не меняются. `force rename` не входит в scope этой change.

Из-за того, что имена команд могут содержать пробелы, синтаксис должен быть однозначным для двух строковых аргументов. Рекомендуемый вариант: `/traderush team rename <currentName> <newName>`, где `currentName` парсится как Brigadier string с поддержкой кавычек и suggestions существующих команд, а `newName` — как `greedyString`. Тогда однословные имена работают как `/traderush team rename Alpha Omega`, а текущие имена с пробелами можно указать как `/traderush team rename "Red Team" Blue Team`.

### Решение: сервер авторитетен, клиент отправляет только намерения

Клиентский экран не должен менять состояние команд локально. Он показывает snapshot, отправляет payload с намерением (`create`, `join`, `leave`, `delete`, `rename`), сервер валидирует действие через `TeamService`, затем возвращает обновлённый snapshot или ошибку.

Это защищает от устаревшего UI и client-side spoofing, а также сохраняет единую точку сохранения состояния.

Рассмотренные альтернативы:

- Полностью синхронизировать состояние через menu slots/data slots: плохо подходит для списков строк и участников.
- Открывать новый screen после каждого действия: проще, но хуже для UX и даёт больше визуального мерцания.

### Решение: custom screen поверх server-opened menu

Блок должен открывать Minecraft UI через server-side interaction. Для этого используется стандартный путь открытия menu/screen от блока, а фактический экран в client source set рисует список команд, детали выбранной команды, text fields и кнопки.

Menu выполняет роль канала открытия и начальной связи с сервером; списки команд и результаты действий удобнее передавать отдельными payloads/snapshots, потому что это динамические коллекции строк, а не слоты инвентаря.

Рассмотренные альтернативы:

- Использовать только чат/книгу/sign-like UI: отклонено, потому что цель change — уйти от чат-команд и дать полноценный UI.
- Использовать inventory-like контейнер со слотами: не подходит для текстового управления командами.

### Решение: UI использует трёхколоночный layout со scrollable panels

Экран управления командами должен быть организован в три колонки:

1. **Список команд** — все команды в порядке `TeamService.listTeams()`, с вертикальным скроллом при переполнении.
2. **Детали выбранной команды** — количество участников, очки и скроллируемый список участников. Если команда не выбрана, колонка показывает empty state без ошибки.
3. **Действия и создание команды** — сверху действия выбранной команды, снизу всегда доступная форма создания новой команды.

При открытии UI выбранная команда определяется так:

1. если игрок состоит в команде, выбрать текущую команду игрока;
2. иначе, если список команд не пустой, выбрать первую команду из списка;
3. иначе не выбирать команду и показать empty state.

В первой колонке есть два независимых визуальных состояния: текущая команда игрока и выбранная для просмотра команда. Эти состояния могут совпадать, поэтому UI должен отображать их совместимо, например через selected background + отдельный marker/label принадлежности игрока, а не через два конфликтующих фона.

Rename и delete относятся к выбранной команде и остаются видимыми в action area. Для непустых команд эти действия disabled. Если оба действия недоступны из-за наличия участников, UI показывает сгруппированное объяснение вроде: «Команда содержит игроков. Переименование и удаление доступны только для пустых команд.»

Layout должен рассчитывать размеры колонок и scroll areas относительно текущих `screen.width`/`screen.height`, чтобы обычный оконный режим и стандартные GUI scale не ломали экран. Списки должны клиппиться внутри своих колонок и скроллиться, а не перекрывать соседние элементы.

`Refresh` является глобальным действием обновления snapshot, а не частью каталога команд, поэтому его следует вынести из первой колонки в footer/status row под основным трёхколоночным блоком. Последнее сообщение об операции или ошибке должно отображаться справа от `Refresh`, чтобы refresh/status воспринимались как связанная служебная область экрана. Освободившееся место в первой колонке следует отдать списку команд: высота списка должна доходить до нижнего внутреннего padding колонки, а не до кнопки `Refresh`.

Форма `Create Team` должна оставаться в нижней части третьей колонки, но не быть прижата к самому низу action viewport. Нужно оставить небольшой нижний запас и поднять форму примерно на 10–15 px относительно текущего положения, чтобы action scrollbar появлялся только при реальном переполнении, а не из-за bottom anchoring/padding, когда элементы фактически помещаются.

### Решение: ограничить разрушающие операции редактирования пустыми командами

В первой версии UI не вводит систему владельцев/капитанов. Чтобы снизить риск griefing и случайных destructive действий, UI позволяет удалять и переименовывать только пустые команды. `force delete` не показывается в UI.

Рассмотренные альтернативы:

- Разрешить всем редактировать любые команды: слишком рискованно.
- Сразу добавить капитанов/owners: правильнее для долгосрочной игры, но заметно расширяет доменную модель и scope.
- Ограничить edit/delete только операторами: безопасно, но противоречит идее player-facing блока управления командами.

### Решение: блок является системным spawn/admin block

Фича должна зарегистрировать block item, чтобы блок можно было выдать/поставить администратором или через creative/admin workflow. Crafting recipe не является частью первой версии, потому что vision описывает блок на спавне как системный объект.

Рассмотренные альтернативы:

- Добавить recipe сразу: преждевременно, пока не определён survival-balance.
- Не добавлять item: усложнит размещение и тестирование блока.

## Риски / Компромиссы

- [Детали Minecraft/Fabric API могут отличаться между версиями] → При реализации проверить актуальные names/mappings для Minecraft `26.1.2`, особенно для menu/screen/networking API.
- [UI snapshot может устареть после изменений от другого игрока] → После каждого успешного действия возвращать обновлённый snapshot; при открытии screen всегда запрашивать актуальное состояние.
- [Griefing через delete/rename] → В первой версии разрешать delete/rename только для пустых команд и не exposing `force delete`.
- [Дублирование отображения ошибок между чат-командами и UI] → Использовать общий `TeamMessages` или вынести message mapping так, чтобы UI и команды использовали один source of truth.
- [Переименование команды требует корректного обновления индекса `teamsByName`] → Реализовать rename через repository-safe путь: удалить/обновить старый name index и сохранить team без потери id/players/score.
- [Трёхколоночный custom UI может стать тесным в оконном режиме] → Рассчитывать размеры колонок от текущего размера экрана, клиппить списки внутри scroll areas и проверять обычные GUI scale/windowed размеры вручную.

## План миграции

1. Добавить block/item/menu/network/screen без изменения existing save format.
2. Если добавляется rename, сохранить текущую модель `Team`: меняется только `name`, id/players/score остаются прежними, format snapshot не требует миграции.
3. Существующие сохранения команд должны загрузиться без изменений.
4. Rollback: удаление блока/UI кода не должно повреждать сохранённые команды, потому что состояние команд хранится отдельно от блока.
