## ADDED Requirements

### Requirement: Регистрация блока управления командами
Система MUST предоставлять отдельный Trade Rush блок управления командами и предмет блока, который администратор или creative/admin workflow может разместить в мире.

#### Scenario: Блок доступен как предмет мода
- **WHEN** мод инициализируется
- **THEN** блок управления командами и его предмет блока зарегистрированы в пространстве имён Trade Rush

#### Scenario: Блок не является vanilla command block
- **WHEN** блок упоминается в коде, assets или тексте UI
- **THEN** он ДОЛЖЕН быть представлен как Trade Rush блок управления командами, а не как vanilla Minecraft command block

### Requirement: Взаимодействие с блоком открывает UI команд
Система MUST открывать UI управления командами, когда игрок взаимодействует с блоком управления командами.

#### Scenario: Игрок открывает UI через блок
- **WHEN** server-side игрок использует блок управления командами
- **THEN** система открывает UI управления командами для этого игрока

#### Scenario: UI открывается с текущим состоянием команд
- **WHEN** UI открыт
- **THEN** UI отображает snapshot, основанный на текущем server-side состоянии команд

### Requirement: Трёхколоночная структура UI
UI управления командами MUST организовывать основной экран в три функциональные колонки: список команд, детали выбранной команды и действия управления.

#### Scenario: UI отображает три функциональные колонки
- **WHEN** UI управления командами открыт
- **THEN** первая колонка отображает список всех команд
- **AND** вторая колонка отображает детали выбранной команды или empty state, если команда не выбрана
- **AND** третья колонка отображает действия выбранной команды в верхней части и форму создания команды в нижней части

#### Scenario: UI выбирает текущую команду игрока при открытии
- **WHEN** игрок открывает UI и состоит в команде
- **THEN** UI выбирает эту команду как команду для просмотра
- **AND** строка этой команды отображает и состояние принадлежности игрока, и состояние выбранности

#### Scenario: UI выбирает первую команду, если игрок не состоит в команде
- **WHEN** игрок открывает UI, не состоит ни в какой команде, и список команд не пустой
- **THEN** UI выбирает первую команду из списка команд для просмотра

#### Scenario: UI корректно открывается без команд
- **WHEN** игрок открывает UI и список команд пуст
- **THEN** UI не выбирает команду для просмотра
- **AND** вторая колонка показывает empty state о том, что команда не выбрана
- **AND** UI не показывает ошибку

### Requirement: Отображение обзора команд
UI управления командами MUST отображать доступные команды и детали, необходимые для выбора команды.

#### Scenario: Отображение списка команд
- **WHEN** UI открыт
- **THEN** первая колонка отображает все команды в порядке, согласованном с `TeamService.listTeams()`
- **AND** каждая строка команды содержит имя команды
- **AND** строка команды MAY дополнительно показывать только компактное количество участников в скобках, например `Red Team (4)`
- **AND** строка команды MUST NOT показывать очки или развёрнутые подписи вроде `participants`/`points`, потому что эти детали отображаются в колонке выбранной команды

#### Scenario: Список команд скроллится при большом количестве команд
- **WHEN** количество команд превышает доступную высоту первой колонки
- **THEN** UI предоставляет вертикальную прокрутку списка команд
- **AND** игрок может выбрать любую команду из списка, включая команды за пределами начальной видимой области

#### Scenario: Refresh не уменьшает каталог команд
- **WHEN** UI отображает первую колонку со списком команд
- **THEN** кнопка `Refresh` расположена вне трёхколоночной области как глобальное действие экрана
- **AND** высота списка команд рассчитывается до нижнего внутреннего padding первой колонки, а не до кнопки `Refresh`
- **AND** последнее status/result сообщение отображается справа от `Refresh` в footer/status row

#### Scenario: Текущая команда игрока выделяется в списке
- **WHEN** игрок состоит в команде
- **THEN** строка этой команды в списке команд имеет визуальное выделение принадлежности игрока

#### Scenario: Выбранная для просмотра команда выделяется в списке
- **WHEN** игрок выбирает команду для просмотра
- **THEN** строка этой команды имеет визуальное выделение выбранного состояния

#### Scenario: Выделения принадлежности и выбранности могут совпадать
- **WHEN** выбранная команда является текущей командой игрока
- **THEN** UI отображает оба состояния так, чтобы они были различимы и не ломали читаемость строки

#### Scenario: Отображение участников выбранной команды
- **WHEN** игрок выбирает команду в UI
- **THEN** UI отображает участников выбранной команды, используя разрешимые имена игроков, где они доступны, и стабильные идентификаторы игроков в остальных случаях

#### Scenario: Отображение текущей команды игрока
- **WHEN** UI открыт
- **THEN** UI показывает, состоит ли игрок сейчас в команде, и указывает эту команду, если она есть

#### Scenario: Вторая колонка показывает сведения выбранной команды
- **WHEN** игрок выбрал команду для просмотра
- **THEN** вторая колонка показывает количество участников и очки выбранной команды
- **AND** ниже отображает список участников выбранной команды

#### Scenario: Список участников скроллится
- **WHEN** количество участников выбранной команды превышает доступную высоту списка участников
- **THEN** UI предоставляет вертикальную прокрутку списка участников
- **AND** игрок может просмотреть всех участников выбранной команды

#### Scenario: Команда для просмотра не выбрана
- **WHEN** нет выбранной команды для просмотра
- **THEN** вторая колонка показывает empty state о том, что команда не выбрана

### Requirement: Создание команды из UI
UI управления командами MUST позволять игроку создать команду с использованием существующих правил валидации имени команды.

#### Scenario: Форма создания команды всегда доступна
- **WHEN** UI управления командами открыт
- **THEN** нижняя часть третьей колонки отображает текстовое поле для названия команды и кнопку создания
- **AND** форма создания доступна независимо от того, выбрана ли команда для просмотра

#### Scenario: Создание валидной команды
- **WHEN** игрок отправляет через UI валидное и уникальное имя команды
- **THEN** сервер создаёт команду через `TeamService`
- **AND** UI обновляется и показывает новую команду
- **AND** поле ввода имени новой команды очищается после успешного создания

#### Scenario: Отклонение невалидного имени команды
- **WHEN** игрок отправляет через UI невалидное имя команды
- **THEN** сервер отклоняет операцию
- **AND** UI показывает игроку понятное сообщение об ошибке

#### Scenario: Отклонение дубликата имени команды
- **WHEN** игрок отправляет имя команды, которое уже существует
- **THEN** сервер отклоняет операцию
- **AND** UI показывает игроку понятное сообщение об ошибке

### Requirement: Вступление в команду и выход из команды через UI
UI управления командами MUST позволять игроку вступить в выбранную команду и выйти из текущей команды через server-side операции команд.

#### Scenario: Вступление в выбранную команду
- **WHEN** игрок выбирает существующую команду и выбирает действие вступления в неё
- **THEN** сервер добавляет игрока в эту команду через `TeamService`
- **AND** UI обновляется и отражает новую команду игрока

#### Scenario: Переход из одной команды в другую
- **WHEN** игрок, уже состоящий в другой команде, вступает в выбранную команду
- **THEN** сервер переносит игрока из старой команды в выбранную команду согласно поведению `TeamService`
- **AND** UI обновляет обе затронутые команды

#### Scenario: Выход из текущей команды
- **WHEN** игрок, который сейчас состоит в команде, выбирает действие выхода
- **THEN** сервер удаляет игрока из этой команды через `TeamService`
- **AND** UI обновляется и показывает, что у игрока нет текущей команды

#### Scenario: Для текущей выбранной команды показывается выход
- **WHEN** выбранная команда является текущей командой игрока
- **THEN** третья колонка показывает действие выхода из команды
- **AND** не показывает действие присоединения к этой же команде

#### Scenario: Для другой выбранной команды показывается присоединение
- **WHEN** выбранная команда не является текущей командой игрока
- **THEN** третья колонка показывает действие присоединения к выбранной команде

#### Scenario: Присоединение к другой команде обновляет выделения
- **WHEN** игрок состоит в одной команде и через UI присоединяется к другой выбранной команде
- **THEN** сервер переносит игрока в выбранную команду через `TeamService`
- **AND** UI обновляет выделение текущей команды игрока и выбранной команды

#### Scenario: Отклонение выхода без текущей команды
- **WHEN** игрок, который не состоит в команде, выбирает действие выхода
- **THEN** сервер отклоняет операцию
- **AND** UI показывает игроку понятное сообщение об ошибке

### Requirement: Удаление пустой команды через UI
UI управления командами MUST позволять удаление пустых команд и MUST NOT предоставлять force deletion непустых команд.

#### Scenario: Удаление выбранной пустой команды
- **WHEN** игрок выбирает команду без участников и выбирает действие удаления
- **THEN** сервер удаляет команду через `TeamService` без force deletion
- **AND** UI обновляется, и команда больше не отображается

#### Scenario: Удаление непустой команды отображается как недоступное
- **WHEN** игрок выбирает команду с одним или несколькими участниками
- **THEN** действие удаления отображается как недоступное
- **AND** UI объясняет, что удаление доступно только для пустых команд

#### Scenario: Отклонение удаления непустой команды сервером
- **WHEN** сервер получает запрос удаления команды с одним или несколькими участниками без force deletion
- **THEN** сервер отклоняет операцию
- **AND** UI показывает игроку понятное сообщение об ошибке, если запрос пришёл из UI

#### Scenario: Force delete недоступен в пользовательском UI
- **WHEN** UI управления командами открыт
- **THEN** он НЕ ДОЛЖЕН предоставлять действие, которое force-deletes команду с участниками

### Requirement: Переименование команды через UI
UI управления командами MUST поддерживать переименование выбранной команды независимо от того, есть ли в ней участники.

#### Scenario: Переименование выбранной команды
- **WHEN** игрок выбирает существующую команду и отправляет валидное, уникальное новое имя
- **THEN** сервер переименовывает команду без изменения её id, очков или участников
- **AND** UI обновляется и показывает новое имя
- **AND** поле ввода нового имени очищается после успешного переименования

#### Scenario: Переименование непустой команды
- **WHEN** игрок выбирает команду с одним или несколькими участниками и отправляет валидное, уникальное новое имя
- **THEN** действие переименования доступно в UI
- **AND** сервер переименовывает команду без изменения её id, очков или участников
- **AND** участники остаются в переименованной команде

#### Scenario: Отклонение невалидного переименования
- **WHEN** игрок отправляет невалидное или дублирующееся новое имя команды
- **THEN** сервер отклоняет операцию
- **AND** UI показывает игроку понятное сообщение об ошибке

### Requirement: Базовая адаптация layout под размер экрана
UI управления командами MUST сохранять читаемость и доступность основных действий при обычном оконном запуске Minecraft и стандартном/default GUI scale. Пользователь MUST NOT быть вынужден уменьшать GUI scale, чтобы получить доступ к основным действиям UI.

#### Scenario: UI открыт в оконном режиме обычного размера при дефолтном GUI scale
- **WHEN** игрок открывает UI в обычном оконном режиме Minecraft с дефолтными настройками GUI scale
- **THEN** три колонки, основные действия и форма создания команды остаются видимыми или доступными через предусмотренные scroll areas
- **AND** третья колонка не допускает перекрытия элементов управления, включая Join/Leave, Rename, Delete, label/text field для имени новой команды и кнопку Create Team
- **AND** если вертикальной высоты третьей колонки недостаточно, её содержимое остаётся доступным через вертикальный scroll или другой предусмотренный layout mechanism
- **AND** списки команд и участников ограничиваются своими колонками и не перекрывают другие элементы

#### Scenario: Третья колонка не показывает ложный scroll
- **WHEN** действия выбранной команды и форма создания команды фактически помещаются в третью колонку
- **THEN** вертикальный scroll третьей колонки не появляется только из-за bottom anchoring или внутренних отступов
- **AND** форма создания команды остаётся в нижней части третьей колонки с небольшим нижним запасом, а не прижатой к самому низу viewport

#### Scenario: Колонки имеют внутренние отступы
- **WHEN** UI отображает три колонки
- **THEN** текст, списки и элементы управления внутри каждой колонки используют небольшой внутренний padding
- **AND** содержимое колонок не прилегает вплотную к границам панелей

#### Scenario: UI пересчитывает layout после изменения размера
- **WHEN** размер окна или GUI scale изменяется и Minecraft пересоздаёт screen layout
- **THEN** UI пересчитывает размеры колонок, списков и элементов управления относительно нового размера экрана
- **AND** выбранная команда и текущие данные snapshot не теряются без необходимости

### Requirement: Server-authoritative действия команд
Система MUST обрабатывать каждое изменяющее команды действие UI на сервере и MUST использовать доменный сервис команд как источник истины.

#### Scenario: Клиент отправляет намерение создать команду
- **WHEN** клиент запрашивает изменяющее команды действие из UI
- **THEN** клиент отправляет на сервер только намерение действия и входные данные
- **AND** сервер валидирует и применяет действие через `TeamService`

#### Scenario: Успешное действие сохраняет состояние
- **WHEN** UI action успешно изменяет состояние команд
- **THEN** тот же путь сохранения, который используется существующими операциями команд, сохраняет обновлённое состояние

#### Scenario: Неуспешное действие не изменяет состояние
- **WHEN** UI action не проходит валидацию или указывает на отсутствующую команду
- **THEN** сервер не изменяет состояние команд
- **AND** UI показывает игроку понятное сообщение об ошибке

### Requirement: Совместимость с существующими командами
Система MUST сохранить существующие чат-команды команд при добавлении UI через блок и MUST expose доменную операцию переименования через chat-command layer, чтобы UI и чат-команды оставались согласованными.

#### Scenario: Команды управления командами остаются зарегистрированными
- **WHEN** мод запускается после этой change
- **THEN** существующие команды `/traderush team` и `/tr team` остаются доступными с текущим поведением

#### Scenario: Чат-команда переименовывает существующую команду
- **WHEN** command source выполняет `/traderush team rename <currentName> <newName>` или `/tr team rename <currentName> <newName>` для существующей команды и валидного уникального нового имени
- **THEN** команда вызывает `TeamService.renameTeam`
- **AND** команда переименовывается без изменения id, очков или участников
- **AND** command source получает success message переименования

#### Scenario: Чат-команда переименования поддерживает имена с пробелами
- **WHEN** текущее имя команды содержит пробелы
- **THEN** command source может указать текущее имя через однозначный quoted string синтаксис
- **AND** новое имя может содержать пробелы и проходит те же trim/length/unique правила, что и создание команды

#### Scenario: Чат-команда отклоняет невалидное переименование
- **WHEN** command source выполняет rename для отсутствующей команды, невалидного нового имени или дублирующегося имени
- **THEN** команда отклоняет операцию через общий `TeamError`/`TeamMessages` mapping
- **AND** состояние команд не изменяется
